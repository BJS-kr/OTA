# Smart Commit

staged 또는 변경된 파일을 분석하여 한국어 Conventional Commits 메시지를 생성합니다.

## 절차

1. **변경 사항 분석**
   - `git diff --cached` (staged 변경) 확인
   - staged가 없으면 `git diff` (unstaged 변경) 확인
   - 변경된 파일 목록과 diff 내용을 분석

2. **커밋 타입 결정**

| Type | 설명 |
|------|------|
| feat | 새로운 기능 추가 |
| fix | 버그 수정 |
| docs | 문서 수정 |
| style | 코드 포맷팅, 세미콜론 누락 등 (비즈니스 로직 변경 없음) |
| refactor | 코드 리팩토링 (기능 변경 없음) |
| test | 테스트 코드 추가 또는 수정 |
| chore | 빌드 업무 수정, 패키지 매니저 설정 등 |

3. **주제(scope) 결정**
   - 변경된 코드의 도메인/기능 영역을 파악
   - 대괄호로 감싸서 맨 앞에 배치: `[주제]`

4. **커밋 메시지 작성**

## 형식

```
[주제] type: 간결한 한국어 설명
```

필요시 본문 추가:
```
[주제] type: 간결한 한국어 설명

- 상세 변경 내용 1
- 상세 변경 내용 2
```

## 좋은 예시

```
[로그인] feat: 사용자 로그인 API 추가
[결제] fix: 결제 페이지 크래시 문제 해결
[문서] docs: README.md 설치 가이드 업데이트
[공통] style: 코드 포맷팅 (Prettier 적용)
[유틸] refactor: 중복된 유틸리티 함수 제거
[인증] test: JWT 토큰 검증 단위 테스트 추가
[빌드] chore: ESLint 설정 업데이트
```

## 나쁜 예시

```
Update code (너무 모호함)
Add login feature (영어로 작성됨)
Login stuff (형식 위반)
feat: added something (영어, 과거형 사용)
```

## 필수 규칙

- **반드시 한국어**로 작성. 예외 없음
- **[주제]** 태그 필수. 변경된 기능/도메인 영역을 한국어로 표기
- **간결하게** 작성. subject는 50자 이내 권장
- **실제 diff 기반**으로 작성. 추측하지 않음
- 여러 영역이 변경된 경우 가장 핵심적인 변경을 기준으로 작성
- 변경이 너무 많으면 커밋을 나누도록 제안

## 실행

1. git diff 분석
2. 커밋 메시지 초안 제시
3. 사용자 확인 후 커밋 실행
